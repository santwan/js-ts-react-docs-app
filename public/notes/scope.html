01 — What is Scope?

Scope is the Rulebook for Where Variables Exist

Scope determines where a variable can be accessed in your code. If a variable is "in scope", you can read and write it. If it's "out of scope", it doesn't exist from where you're standing — trying to access it throws a ReferenceError.

JavaScript has three types of scope: global scope, function scope, and block scope. Understanding which scope a variable lives in is fundamental to avoiding bugs and writing clean code.

💡

Think of scope like a building with floors. Variables on a higher floor can see everything below them — but variables on a lower floor can't see what's above. Code inside a function can see variables outside it. The reverse is never true.

02 — The Three Types of Scope

Global Scope

Variables declared outside any function or block live in the global scope. They're accessible from anywhere in your code. In a browser, global variables become properties of the window object.

Global variables sound convenient — but they cause bugs in large codebases because any piece of code can accidentally overwrite them. Minimise globals.

const appName = "MyApp"; // global scope

function greet() {
  console.log(appName); // ✓ accessible — inner can see outer
}

greet(); // "MyApp"

Function Scope

Variables declared inside a function are only accessible within that function. They are created when the function is called and destroyed when it returns. This is true for var, let, and const — all three are function-scoped in this sense (though let/const are also block-scoped).

function calculateTotal() {
  const tax = 0.18; // function scope — only lives here
  const price = 100;
  return price * (1 + tax);
}

console.log(tax); // ReferenceError: tax is not defined
// tax doesn't exist outside the function
💡

Function scope is like a hotel room — what happens inside stays inside. Variables created in a function are private to that function. The corridor (outer scope) has no idea they exist.

Block Scope

A block is any code wrapped in curly braces {} — an if statement, a for loop, a while loop, or even a standalone {}. let and const respect block scope. var does NOT — this is var's biggest problem.

⚠ var ignores blocks
for (var i = 0; i < 3; i++) {
  // var leaks out of the loop
}
console.log(i); // 3 — var leaked! 😱

if (true) {
  var secret = "exposed";
}
console.log(secret); // "exposed" 😱
✓ let/const respect blocks
for (let i = 0; i < 3; i++) {
  // i stays inside the loop
}
console.log(i); // ReferenceError ✓

if (true) {
  let secret = "contained";
}
console.log(secret); // ReferenceError ✓
03 — Lexical Scope

Scope is Determined by Where You Write the Code, Not Where You Call It

JavaScript uses lexical scoping (also called static scoping). This means a function's scope is determined by where it was defined in the source code — not where it was called from. This is a crucial distinction.

💡

"Lexical" comes from "lexicon" — the written text. Your scope is baked in at the time you write the code. A function carries its birth environment with it wherever it goes — even if it's called from a completely different place.

const city = "Kolkata"; // defined in global scope

function inner() {
  console.log(city); // ✓ sees city — it was defined in the outer scope
}

function outer() {
  const city = "Mumbai"; // a different city, in outer's scope
  inner();               // calling inner from here...
}

outer(); // logs "Kolkata" — NOT "Mumbai"!
// inner was DEFINED in global scope, so it sees global's city
// it doesn't matter that it was CALLED from inside outer
⚡

This surprises people every time. inner() logs "Kolkata" not "Mumbai" because lexical scope is about definition location, not call location. This is also the foundation of closures (Topic 16).

04 — The Scope Chain

How JavaScript Looks Up Variables

When you reference a variable, JavaScript doesn't just look in the current scope and give up. It performs a scope chain lookup — starting in the current scope and walking outward, one level at a time, until it either finds the variable or reaches the global scope and throws a ReferenceError.

💡

Imagine nested boxes — a small box inside a medium box inside a large box. When JS looks for a variable, it checks the innermost box first. If not found, it checks the next box out. Then the next. If it reaches the outermost box and still can't find it — ReferenceError. It never looks inward from the outside.

const a = "global";          // level 1 — global

function outer() {
  const b = "outer";          // level 2 — outer's scope

  function inner() {
    const c = "inner";        // level 3 — inner's scope

    console.log(c); // ✓ found at level 3
    console.log(b); // ✓ not at 3, found at level 2
    console.log(a); // ✓ not at 3 or 2, found at level 1
    console.log(d); // ✗ not at 3, 2, or 1 — ReferenceError
  }

  inner();
  console.log(c); // ✗ outer can't see inner's variables
}

outer();
Global Scope
a = "global" outer (function)
outer() — Function Scope
b = "outer" inner (function) ↑ can also see: a
inner() — Function Scope
c = "inner" ↑ can also see: b, a d (not defined anywhere)
05 — Variable Shadowing

Inner Scope Can Redeclare a Name From Outer Scope

When you declare a variable in an inner scope with the same name as one in an outer scope, the inner variable shadows the outer one. Inside that inner scope, you always get the inner variable — the outer one is temporarily invisible, but not destroyed.

const message = "Hello from global";

function greet() {
  const message = "Hello from function"; // shadows the global
  console.log(message); // "Hello from function"

  if (true) {
    const message = "Hello from block"; // shadows the function-level one
    console.log(message); // "Hello from block"
  }

  console.log(message); // "Hello from function" — back to function level
}

greet();
console.log(message); // "Hello from global" — untouched all along
⚡

Shadowing is legal but often a mistake. Using the same variable name at multiple scope levels makes code confusing to read and debug. If you find yourself shadowing, it usually means the inner variable needs a better name. One common exception: function parameters named error in catch blocks.

06 — Interview Q&A
Interview Questions you must be able to answer out loud
What is scope in JavaScript?
Scope defines where a variable can be accessed. JavaScript has three kinds: global scope (accessible everywhere), function scope (accessible only inside the function it was declared in), and block scope (accessible only inside the {} it was declared in — for let and const only).
What is lexical scope?
Lexical scope means a function's scope is determined by where it was defined in the source code, not where it was called from. When JS looks up a variable inside a function, it walks up through the scopes that existed at the point the function was written — not the scopes that exist at the point it was called.
What is the scope chain?
When JavaScript can't find a variable in the current scope, it walks up to the parent scope, then the grandparent scope, and so on — all the way to the global scope. This chain of nested scopes is the scope chain. If the variable isn't found anywhere in the chain, JavaScript throws a ReferenceError. Variables can always see outward, never inward.
What is the difference between function scope and block scope?
Function scope means a variable is accessible anywhere within the function it was declared in. Block scope means a variable is only accessible within the specific {} block it was declared in. let and const are block-scoped. var is only function-scoped — it leaks out of if statements, loops, and any block that isn't a function.
Why is var's function scope a problem?
Because var ignores blocks like if statements and for loops. A loop counter declared with var leaks into the surrounding function — which means it can accidentally overwrite other variables or be read when it shouldn't be. This is one of the main reasons let was introduced.
What is variable shadowing?
Shadowing is when a variable in an inner scope has the same name as one in an outer scope. Inside the inner scope, the outer variable becomes invisible — the inner one "shadows" it. The outer variable still exists and is unchanged; it just can't be reached from inside the shadow. It's legal but usually a sign the inner variable needs a more descriptive name.
🔷

TypeScript Layer — Scope rules are identical in TypeScript. What TS adds is the ability to catch scope-related type errors at compile time, and a few patterns (like module scope) that become much more important in typed codebases.

07 — How TypeScript Handles Scope

TypeScript Tracks Types Scope-by-Scope

The scope rules you just learned are exactly the same in TypeScript — TypeScript is built on JavaScript's scoping model. What TypeScript adds is that it tracks the type of every variable at every scope level. If you redeclare a variable name in a nested scope with a different type, TypeScript knows about it.

const value: number = 42;       // global: value is number

function process() {
  const value: string = "hello"; // shadows — now value is string here
  console.log(value.toUpperCase()); // ✓ TS knows it's a string
}

// Back in global scope — value is still number
value.toUpperCase(); // TS Error: number has no toUpperCase method
💡

TypeScript's type checker runs scope-aware analysis. It doesn't just know that a variable exists — it knows what type it has at every point in your code. This is why TypeScript can catch errors that plain linters miss.

08 — Type Narrowing & Scope

TypeScript Narrows Types Inside Blocks

This is one of TypeScript's most powerful features, and it's built entirely on block scope. When you check a type inside an if block, TypeScript narrows the type within that block — it knows more about the variable inside the block than outside it. This is called control flow analysis.

function printLength(value: string | number) {
  // Here: value is string | number

  if (typeof value === "string") {
    // Inside this block: value is narrowed to string
    console.log(value.length);       // ✓ string has .length
    console.log(value.toUpperCase()); // ✓ string has .toUpperCase
  } else {
    // Inside this block: value is narrowed to number
    console.log(value.toFixed(2));   // ✓ number has .toFixed
    console.log(value.length);       // TS Error: number has no .length
  }

  // Back here: value is string | number again
}
💡

TypeScript uses block scope to track type knowledge. Before the if-block: unknown which. Inside the if-block: TypeScript knows exactly which. After the if-block: back to uncertain. The type information flows with the scope.

Narrowing Persists After a Guard

If you return or throw inside a block, TypeScript is smart enough to narrow the type for everything that comes after — because it knows the other possibility was eliminated. This pattern is called an early return guard and is a very common, clean TS pattern.

function getUpperCase(value: string | null): string {
  // Guard — handle the null case and exit early
  if (value === null) {
    return "";
  }

  // From here down — TS knows value is string, not null
  return value.toUpperCase(); // ✓ no need for null check
}
09 — Module Scope in TypeScript

Every TypeScript File is Either a Script or a Module

This distinction matters a lot in practice. A script has no imports or exports — all its declarations go into the global scope, which means naming collisions across files are possible. A module has at least one import or export — all its declarations are module-scoped, meaning they're private to that file unless explicitly exported.

In modern TypeScript (and all real projects), every file should be a module. This avoids global scope pollution entirely.

// user.ts — this is a MODULE (has an export)
const MAX_NAME_LENGTH = 50; // module-scoped — private to this file

export interface User {
  name: string;
  age: number;
}

export function createUser(name: string, age: number): User {
  return { name, age };
}

// app.ts — importing from user.ts
import { User, createUser } from "./user";

const me: User = createUser("Arjun", 25);
console.log(MAX_NAME_LENGTH); // Error — not exported, module-private
💡

Module scope is like scope at the file level. Anything not exported is completely private — no other file can see it. export is you deliberately punching a hole in that privacy. This is the cleanest scoping model in JS/TS and the one you'll use in all real projects.

10 — TypeScript Interview Q&A
Interview TypeScript questions on this topic
How does TypeScript use scope for type checking?
TypeScript runs a scope-aware analysis called control flow analysis. It tracks the type of every variable at every point in the code — including inside blocks. When you check a type condition in an if-block, TypeScript narrows the variable's type within that block. It knows more inside the block than it does outside it.
What is type narrowing?
Type narrowing is when TypeScript reduces a broader type to a more specific one based on a runtime check. For example, if a parameter is typed as string | number, doing typeof value === "string" inside an if-block narrows the type to just string within that block. TypeScript uses block scope to know exactly when that narrowing applies.
What is the difference between a TypeScript script and a module?
A script has no import or export statements — its variables are in the global scope and can clash with other files. A module has at least one import or export — its declarations are file-scoped and private unless explicitly exported. All real TS projects use modules. If two files both declare the same variable name, there's no conflict as long as both are modules.
What is an early return guard and why is it useful in TypeScript?
An early return guard is an if-check at the top of a function that handles an edge case (like null) and returns early. After that check, TypeScript knows the edge case has been eliminated — so the rest of the function works with the narrowed type, no optional chaining or extra checks needed. It keeps code flat and readable while satisfying TypeScript's type safety requirements.
11 — Key Takeaways

JavaScript + TypeScript — What to Lock In

  • Scope controls where variables are accessible. Three kinds: global, function, block.
  • let and const are block-scoped. var is only function-scoped — it leaks out of if/for/while blocks. This is why var is avoided.
  • Lexical scope means scope is determined by where you write the code, not where you call it.
  • The scope chain is JavaScript's lookup process — it searches inward-to-outward. Inner scopes can see outer, never the reverse.
  • Variable shadowing is legal but usually a code smell. Same name at multiple levels creates confusion — rename the inner one.
  • TypeScript does control flow analysis — it tracks types scope-by-scope and narrows them inside conditional blocks.
  • Type narrowing is one of TS's most useful features — use typeof, instanceof, and equality checks to narrow union types inside blocks.
  • The early return guard pattern is the cleanest way to handle null/undefined in TypeScript — check at the top, return early, then work with the clean type below.
  • In TypeScript, every file should be a module — use import/export to keep declarations out of global scope.
