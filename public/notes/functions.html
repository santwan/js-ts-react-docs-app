Functions —
All Forms

JS Core TypeScript Interview Topic Practical
01 — Why This Topic Matters

Functions Are the Core Unit of JavaScript

Everything in JavaScript eventually comes back to functions. Callbacks, event handlers, array methods, promises, React components, custom hooks — they are all functions. Understanding the different forms a function can take — and crucially, how their behaviour differs — is not optional knowledge. It is the foundation everything else sits on.

There are four forms to know: function declarations, function expressions, arrow functions, and methods. They all create callable functions — but they differ in hoisting, this binding, the arguments object, and how they can be used.

💡

The most important difference between function forms is how they bind this. Arrow functions don't have their own this — they inherit it from wherever they were written. Regular functions get a new this every time they're called. This single difference explains most of the "why isn't this what I expected?" bugs.

02 — Function Declarations

The Classic, Named, Fully Hoisted Form

A function declaration uses the function keyword as a statement — not inside an expression. It is fully hoisted (see Topic 03), has its own this, its own arguments object, and can be used as a constructor with new.

// Syntax
function greet(name) {
  return "Hello, " + name;
}

// Can be called before it appears — fully hoisted
greet("Arjun"); // "Hello, Arjun" ✓

function greet(name) {
  return "Hello, " + name;
}

// Default parameters (ES6+)
function greet(name = "stranger") {
  return "Hello, " + name;
}

greet();          // "Hello, stranger"
greet("Arjun");   // "Hello, Arjun"

// Rest parameters — collects remaining args into an array
function sum(first, ...rest) {
  return rest.reduce((acc, n) => acc + n, first);
}

sum(1, 2, 3, 4); // 10
03 — Function Expressions

A Function Assigned to a Variable

A function expression is a function created inside an expression — most commonly by assigning it to a variable. It is not hoisted in the usable sense. It can be named (useful for stack traces) or anonymous.

Behaviourally, a regular function expression is identical to a function declaration — same this rules, same arguments object. The difference is purely about hoisting and where it can appear in code.

// Anonymous function expression
const greet = function(name) {
  return "Hello, " + name;
};

// Named function expression — the name is only visible inside the function
// Useful for recursion and better stack traces
const factorial = function fact(n) {
  if (n <= 1) return 1;
  return n * fact(n - 1); // can call itself by name inside
};

factorial(5); // 120
fact(5);      // ReferenceError — 'fact' not visible outside
💡

Function expressions are great for passing functions as arguments (callbacks), returning functions from functions, and conditional function definitions — cases where you need a function to be a value, not a statement.

04 — Arrow Functions

Shorter Syntax — And a Fundamentally Different this

Arrow functions (=>) were introduced in ES6. They are not just shorthand — they behave differently from regular functions in important ways. The key differences are: they have no own this, no arguments object, cannot be used as constructors, and cannot be used as generators.

// Various arrow function forms
const double  = n => n * 2;           // single param, implicit return
const add     = (a, b) => a + b;       // multiple params
const greet   = () => "Hello!";        // no params
const getUser = () => ({ name: "Arjun" }); // return object — wrap in ()

// Multi-line — needs explicit return
const process = (data) => {
  const cleaned = data.trim();
  return cleaned.toUpperCase();
};
⚡

Returning an object literal needs parentheses. () => {} looks like a function body, not an object — JS will interpret {} as a block. Wrap the object in (): () => ({ key: value }). This trips up React developers constantly when writing map callbacks.

Arrow Functions & this — The Critical Difference

Regular functions get their own this — determined at call time by how they were called. Arrow functions do not get their own this — they capture this from the enclosing scope at the time they were defined. This is called lexical this.

⚠ Regular function loses this
const timer = {
  count: 0,
  start() {
    setInterval(function() {
      this.count++; // 'this' is window!
      // not the timer object 😱
    }, 1000);
  }
};
✓ Arrow function keeps this
const timer = {
  count: 0,
  start() {
    setInterval(() => {
      this.count++; // 'this' is timer ✓
      // inherited from start()
    }, 1000);
  }
};
💡

Arrow functions "freeze" this at the moment they're written — like taking a snapshot. Regular functions look up this fresh every time they're called. Use arrow functions for callbacks inside methods — they'll carry the right this with them.

⚡

Never use arrow functions as object methods. Since arrow functions inherit this from the enclosing scope (the outer scope, likely window or undefined), using them as a method means this won't be the object. Use regular method shorthand for object methods.

05 — Methods (Functions in Objects)

A Method is Just a Function That Lives in an Object

When a function is a property of an object, it's called a method. ES6 introduced a clean shorthand syntax. When a method is called as obj.method(), this inside it refers to obj.

// Old way — function expression as property
const user = {
  name: "Arjun",
  greet: function() {
    return "Hi, I'm " + this.name;
  }
};

// ES6 shorthand — cleaner, preferred
const user = {
  name: "Arjun",
  greet() {                        // method shorthand
    return "Hi, I'm " + this.name;
  }
};

user.greet(); // "Hi, I'm Arjun" — this = user ✓

// Detaching a method loses this
const fn = user.greet;
fn(); // "Hi, I'm undefined" — this is now window/undefined
⚡

Detaching a method loses its this. When you do const fn = obj.method and then call fn(), it's no longer called as a method of obj — so this is lost. This is exactly the bug that bind() was created to solve (Topic 05).

06 — IIFE (Immediately Invoked Function Expression)

A Function That Runs Itself Immediately

An IIFE is a function that is defined and called in the same expression. It was the primary pattern for creating private scope before ES modules existed. You'll encounter it in legacy code and it still appears occasionally in modern code for initialisation logic.

// Classic IIFE syntax
(function() {
  const privateVar = "only here"; // can't be accessed outside
  console.log("I ran immediately");
})();

// Arrow function IIFE
(() => {
  console.log("Also runs immediately");
})();

// IIFE with a return value
const config = (() => {
  const env = "production";
  return { env, debug: false };
})();

console.log(config.env); // "production"
💡

IIFEs solve the same problem that modules solve now — keeping variables private and avoiding global scope pollution. In modern JS/TS with ESM modules, you rarely need IIFEs. But you'll see them in older codebases and bundler output, so recognising the pattern matters.

08 — Interview Q&A
Interview Questions you must be able to answer out loud
What is the difference between a function declaration and a function expression?
A function declaration uses the function keyword as a statement and is fully hoisted — you can call it anywhere in its scope. A function expression assigns a function to a variable and follows that variable's hoisting rules — it's not usable before its line. Behaviourally (for this, arguments, etc.) they behave the same. The difference is hoisting and where they can be written.
What is the difference between arrow functions and regular functions?
Four key differences: (1) Arrow functions have no own this — they inherit it from the enclosing scope (lexical this). (2) They have no arguments object — use rest params instead. (3) They cannot be used as constructors with new. (4) They cannot be generators. These aren't shortcomings — they're deliberate design choices that make arrow functions safer to use as callbacks.
When would you use an arrow function vs a regular function?
Use arrow functions for callbacks, array method callbacks, event handlers, and anything where you need to inherit this from the surrounding context. Use regular functions (or method shorthand) for object methods, class methods, and when you need the function to have its own this. In modern TypeScript code, most standalone utility functions are arrow functions assigned to const.
What does "lexical this" mean?
Lexical this means the value of this is determined by where the function was written (its lexical position in the code), not by how it was called. Arrow functions use lexical this — they capture this from the enclosing scope at definition time. Regular functions determine this at call time. This makes arrow functions predictable inside callbacks and event handlers.
What is an IIFE and when would you use one?
An IIFE (Immediately Invoked Function Expression) is a function that defines and calls itself in one expression. Its main use is creating a private scope — variables inside it can't pollute the outer scope. Before ES modules, this was the standard pattern for encapsulation. In modern code with ES modules and block scoping, you rarely need them — but you'll encounter them in legacy code and sometimes for inline async initialisation with (async () => { await ... })().
What happens when you detach a method from its object?
The method loses its this binding. When you do const fn = obj.method and call fn(), it's no longer a method call — it's a plain function call, so this becomes the global object (or undefined in strict mode). To fix this, you can use bind() — const fn = obj.method.bind(obj) — to permanently attach the correct this.
🔷

TypeScript Layer — TypeScript adds type annotations to parameters, return values, and the functions themselves. The behavioural differences between function forms are unchanged — TypeScript just makes them type-safe and self-documenting.

09 — Typing Parameters & Return Values

Annotating Functions in TypeScript

In TypeScript, you annotate each parameter and the return value. TypeScript can often infer the return type — but annotating it explicitly is good practice because it documents intent and catches errors when the function body doesn't match what you said it returns.

// Function declaration with full annotations
function add(a: number, b: number): number {
  return a + b;
}

// Arrow function
const multiply = (a: number, b: number): number => a * b;

// Optional parameter — must come after required ones
function greet(name: string, greeting?: string): string {
  return (greeting ?? "Hello") + ", " + name;
}

greet("Arjun");            // "Hello, Arjun"
greet("Arjun", "Hey");     // "Hey, Arjun"

// Default parameter — TS infers type from the default
function createUser(name: string, role: string = "user") {
  return { name, role };
}

// Rest parameters — typed as an array
function logAll(label: string, ...items: string[]): void {
  items.forEach(item => console.log(label, item));
}
10 — Typing Functions as Values

Functions Are Values — They Need Types Too

When you pass a function as an argument, store it in a variable, or return it from another function, TypeScript needs to know its type — what parameters it accepts and what it returns. This is where function type syntax comes in.

// Typing a function parameter that accepts a callback
function runTwice(fn: () => void): void {
  fn();
  fn();
}

// Callback with parameters
function transform(value: number, fn: (n: number) => number): number {
  return fn(value);
}

transform(5, n => n * 2);  // 10 ✓
transform(5, n => "hi");  // Error: string not assignable to number

// Type alias for a function type — cleaner for reuse
type Formatter = (value: string) => string;

const upper: Formatter = s => s.toUpperCase();
const trim:  Formatter = s => s.trim();

// Function returning a function — typed precisely
function multiplier(factor: number): (n: number) => number {
  return n => n * factor;
}

const double = multiplier(2);
double(5); // 10 ✓
💡

Use a type alias for function types you reuse in multiple places. Inline function types ((n: number) => number) are fine for one-off use but get hard to read when the signature is long. Naming it makes the code self-documenting.

11 — void vs never Return Types

Two Special Return Types You Must Know

void means the function intentionally returns nothing — it may return undefined implicitly, but you don't care about the return value. Use this for event handlers and side-effect functions.

never means the function never completes normally — it either always throws, or runs an infinite loop. TypeScript uses this for exhaustive checks.

// void — runs, returns nothing you care about
function logMessage(msg: string): void {
  console.log(msg);
  // implicitly returns undefined — fine for void
}

// never — always throws, never returns
function fail(message: string): never {
  throw new Error(message);
}

// never used for exhaustive checks — TS will error if a case is missed
type Status = "active" | "inactive" | "banned";

function handleStatus(s: Status): string {
  switch (s) {
    case "active":   return "User is active";
    case "inactive": return "User is inactive";
    case "banned":   return "User is banned";
    default:
      return fail("Unhandled status: " + s); // TS: s is never here ✓
  }
}
💡

void = "this function runs but gives you nothing back." never = "this function doesn't come back at all." If you add a new status to the union type and forget to add a case in the switch, TypeScript will error at the never call — because s is no longer never. This pattern makes your code refactor-proof.

12 — TypeScript Interview Q&A
Interview TypeScript questions on this topic
How do you type a function in TypeScript?
You annotate each parameter with its type after a colon, and the return type after the closing parenthesis: function fn(a: number, b: string): boolean. TypeScript can often infer the return type, but annotating it explicitly is good practice — it documents intent and catches mismatches between what you said the function returns and what it actually returns.
What is the difference between void and never as return types?
void means the function returns nothing useful — it completes normally but you don't care about its return value. It's used for side-effect functions like event handlers and loggers. never means the function never completes normally — it always throws an error or runs forever. It's used for functions that are guaranteed to fail and for exhaustive switch-case checks.
How do you type a function that accepts a callback?
You annotate the callback parameter with a function type signature: function run(cb: (value: string) => void). If you reuse the callback type in multiple places, extract it to a type alias: type Handler = (value: string) => void. TypeScript will then enforce that any function passed as a callback matches that signature.
What is the difference between an optional parameter and a default parameter in TypeScript?
An optional parameter (name?: string) may or may not be provided — its type inside the function is string | undefined, so you need to handle both cases. A default parameter (name: string = "stranger") is always a string inside the function because if it's not provided, the default is used. TypeScript infers the type from the default value, so you don't need to annotate it separately.
13 — Key Takeaways

JavaScript + TypeScript — What to Lock In

  • Function declarations are fully hoisted and have their own this. Use them for named utilities and exported functions.
  • Function expressions follow variable hoisting rules. Behaviourally the same as declarations — the difference is hoisting.
  • Arrow functions have no own this — they inherit it lexically. No arguments object. Cannot be used with new.
  • Use arrow functions for callbacks. Use method shorthand for object/class methods. Never use arrow functions as object methods.
  • Returning an object literal from an arrow function requires wrapping in () — otherwise JS reads {} as a function body.
  • Detaching a method from its object loses this — use bind() to fix it (covered in Topic 05).
  • IIFEs create private scope immediately. Rare in modern code but common in legacy codebases and bundler output.
  • In TypeScript, annotate all parameters and return types explicitly. Use void for side-effect functions, never for functions that always throw.
  • Use type aliases for function types you reuse — type Handler = (e: Event) => void is cleaner than repeating the inline type everywhere.
  • Optional params (?) give you T | undefined inside. Default params give you a clean T — the safer, cleaner choice when a sensible default exists.
