The this Keyword

JS Core TypeScript Interview Topic Concept
01 — What is this?

this is a Reference to the Execution Context

this is a special keyword that refers to the object that is currently executing the function. It is not set when you write the function — it is determined at call time, based on how the function is invoked. This is what makes this confusing — the same function can have a different this depending on how it's called.

There are four binding rules that determine what this is, evaluated in order of precedence. Once you know these four rules, this becomes completely predictable.

💡

Think of this as the answer to "who called me?". A function doesn't own this — its caller sets it. If user.greet() calls a function, then this is user. If a bare greet() calls it, this is the global object. Same function, different caller, different this.

Rule 1 — Highest priority
new binding
When called with new, this is the newly created object.
Rule 2
Explicit binding
call(), apply(), bind() — you set this yourself.
Rule 3
Implicit binding
Called as obj.method() — this is the object left of the dot.
Rule 4 — Lowest priority
Default binding
Plain function call — this is window (or undefined in strict mode).
02 — Rule 4: Default Binding (Plain Function Call)

The Fallback — and the Most Surprising

When a function is called as a standalone function — no dot, no new, no call/apply/bind — this defaults to the global object (window in browsers). In strict mode (which TypeScript enforces by default), it's undefined instead.

function showThis() {
  console.log(this);
}

showThis(); // window (browser, non-strict) / undefined (strict mode)

// Strict mode — common source of "Cannot read property of undefined"
"use strict";

function strictThis() {
  console.log(this); // undefined
}

strictThis(); // undefined — and accessing this.anything crashes
⚡

TypeScript (and ES modules) run in strict mode by default. This means plain function calls give you this === undefined, not the global object. Accessing any property on it will throw. This is actually safer — it makes the bug loud and obvious.

03 — Rule 3: Implicit Binding (Method Call)

The Object Left of the Dot Owns this

When a function is called as a method — obj.method() — this is implicitly set to obj. Whatever sits to the left of the last dot at call time becomes this. This is the most common binding you'll work with.

const user = {
  name: "Arjun",
  greet() {
    return "Hello, I'm " + this.name;
  }
};

user.greet(); // "Hello, I'm Arjun" — this = user ✓

// Nested objects — this is always the immediate parent
const company = {
  name: "Acme",
  ceo: {
    name: "Alice",
    introduce() {
      return "I'm " + this.name; // this = company.ceo (the immediate dot-owner)
    }
  }
};

company.ceo.introduce(); // "I'm Alice" — not "I'm Acme"

Implicit Binding Loss — The #1 this Bug

Implicit binding is easily lost. If you extract a method from an object and call it as a plain function, the object connection is severed and this falls back to default binding.

⚠ Binding lost on extraction
const user = {
  name: "Arjun",
  greet() {
    return this.name;
  }
};

const fn = user.greet; // extract
fn(); // undefined 😱
// called as plain fn(), not user.fn()
⚠ Binding lost in callback
const user = {
  name: "Arjun",
  delayedGreet() {
    setTimeout(function() {
      console.log(this.name);
      // this = window 😱
    }, 100);
  }
};

user.delayedGreet();
💡

The rule is simple: if there's no dot at the call site, implicit binding doesn't apply. const fn = obj.method; fn() — the dot is gone by call time. setTimeout(obj.method, 100) — same thing, the function is passed as a reference, no dot at the moment it runs.

04 — Rule 2: Explicit Binding (call, apply, bind)

You Take Control of this Yourself

JavaScript gives you three methods — call(), apply(), and bind() — that let you explicitly set what this will be when a function runs. All three are methods on the Function prototype — meaning every function in JavaScript has them.

function greet(greeting, punctuation) {
  return greeting + " " + this.name + punctuation;
}

const user = { name: "Arjun" };

// call — invoke immediately, args passed individually
greet.call(user, "Hello", "!");    // "Hello Arjun!"

// apply — invoke immediately, args passed as array
greet.apply(user, ["Hello", "!"]);  // "Hello Arjun!"

// bind — returns a NEW function with this permanently locked
const boundGreet = greet.bind(user);
boundGreet("Hey", ".");             // "Hey Arjun."
boundGreet("Hi", "!");              // "Hi Arjun!" — this is always user
💡

call = call now, comma-separated args. apply = call now, array of args. bind = don't call yet, give me a new function with this locked in. In modern code, bind() is the most useful of the three — it's how you fix detached methods and pass methods as callbacks while preserving this.

bind() in Practice — Fixing Detached Methods

class Counter {
  constructor() {
    this.count = 0;
    // Fix: bind at construction time so increment always has the right this
    this.increment = this.increment.bind(this);
  }

  increment() {
    this.count++;
    console.log(this.count);
  }
}

const counter = new Counter();
const btn = document.querySelector("button");
btn.addEventListener("click", counter.increment); // ✓ this is always counter

// Modern alternative: arrow function in class field (same result, cleaner)
class Counter {
  count = 0;
  increment = () => { // arrow function class field — lexical this
    this.count++;
    console.log(this.count);
  };
}
05 — Rule 1: new Binding (Constructor Call)

new Creates a Fresh Object and Binds this to It

When you call a function with new, JavaScript does four things automatically: (1) creates a new empty object, (2) sets this to that object inside the function, (3) links the object's prototype to the function's prototype property, and (4) returns the object (unless the function explicitly returns a different object).

function Person(name, age) {
  // 'this' is the brand new object created by 'new'
  this.name = name;
  this.age  = age;
  // implicitly returns 'this'
}

const arjun = new Person("Arjun", 25);
console.log(arjun.name); // "Arjun"
console.log(arjun.age);  // 25

// new binding has highest precedence — even bind can't override it
const fakeThis = { name: "fake" };
const BoundPerson = Person.bind(fakeThis);
const raj = new BoundPerson("Raj", 30); // new wins — this is the new object
console.log(raj.name); // "Raj" — not "fake"
06 — Arrow Functions: The Exception to All Rules

Arrow Functions Don't Follow Any of the Four Rules

Arrow functions have no own this. They don't create a this binding at all. When you reference this inside an arrow function, JavaScript goes to the enclosing lexical scope and uses that scope's this — whatever it is at the point the arrow function was written. No rules apply, no binding happens. It's purely lexical.

const user = {
  name: "Arjun",
  hobbies: ["coding", "reading"],

  printHobbies() {
    // 'this' here = user (implicit binding, method call)
    this.hobbies.forEach(hobby => {
      // Arrow function — no own 'this', inherits from printHobbies
      // So 'this' here is also 'user' ✓
      console.log(this.name + " likes " + hobby);
    });
  }
};

user.printHobbies();
// "Arjun likes coding"
// "Arjun likes reading"

// You cannot bind a different 'this' to an arrow function — it's ignored
const arrow = () => this;
arrow.call({ name: "fake" }); // still the lexical this — call is ignored
💡

Arrow functions are this-transparent. They pass this through from the outside unchanged. That's why they're perfect for callbacks inside methods — they don't disrupt the outer this. And that's why they're wrong for object methods themselves — they'd inherit from the global scope, not the object.

08 — Interview Q&A
Interview Questions you must be able to answer out loud
What is this in JavaScript?
this is a keyword that refers to the object currently executing the function. Its value isn't determined when you write the function — it's determined at call time, based on how the function is invoked. There are four binding rules, in order of precedence: new binding (highest), explicit binding (call/apply/bind), implicit binding (method call), and default binding (plain function call, lowest).
What is the difference between call, apply, and bind?
All three explicitly set this for a function. call invokes the function immediately, passing arguments individually. apply invokes immediately, passing arguments as an array. bind does not invoke — it returns a new function with this permanently locked to the value you pass. bind is the most commonly used of the three in practice.
Why do arrow functions solve the this problem in callbacks?
Regular functions create their own this binding when called — so a callback passed to setTimeout or forEach gets a new this (usually window or undefined), losing the outer method's this. Arrow functions have no own this — they inherit it from the surrounding scope where they were written. So using an arrow function as a callback inside a method preserves the method's this.
What is implicit binding loss?
Implicit binding loss happens when you extract a method from an object and call it as a plain function. const fn = obj.method; fn() — the dot is gone at call time, so implicit binding doesn't apply and this falls back to the default. It also happens when passing a method as a callback — the function reference is passed without its object context. Fix it with bind() or an arrow function wrapper.
Can you use call or apply to change the this of an arrow function?
No. Arrow functions have no own this — call, apply, and bind are all silently ignored for the this binding. The arrow function will always use the this from its enclosing scope at the time it was defined. This is by design — it's what makes them safe to use as callbacks.
What does new do to this?
When you call a function with new, JavaScript creates a brand new empty object, sets this to that object inside the function, links the object's prototype, and then returns that object. new binding has the highest precedence — it overrides even bind().
🔷

TypeScript Layer — TypeScript makes this safer in two key ways: it can type-check what this refers to inside a function, and it enforces strict mode by default — meaning unsafe default binding gives you undefined instead of silently using the global object.

09 — this in TypeScript Classes

TypeScript Knows the Type of this Inside Classes

Inside a TypeScript class, this is fully typed — TypeScript knows its type is the class itself. This means you get full autocompletion and type checking on everything you access through this. It will also catch cases where you pass a method as a callback and lose this — at compile time.

class UserService {
  private users: string[] = [];

  addUser(name: string): void {
    this.users.push(name); // TS knows this.users is string[] ✓
  }

  getCount(): number {
    return this.users.length; // fully typed ✓
  }
}

const service = new UserService();
service.addUser("Arjun");

// Arrow function class fields — the modern TS pattern for safe this
class Button {
  label: string;

  constructor(label: string) {
    this.label = label;
  }

  // Arrow field — this is always the Button instance, even as a callback
  handleClick = (): void => {
    console.log("Clicked: " + this.label); // always safe ✓
  };
}

const btn = new Button("Submit");
document.addEventListener("click", btn.handleClick); // ✓ this is safe
💡

The arrow function class field pattern (handleClick = () => {}) is the standard TypeScript solution for event handlers and callbacks from classes. It's cleaner than bind(this) in the constructor and fully type-safe. You'll see this pattern everywhere in React class components and TypeScript services.

10 — The this Parameter in TypeScript

Explicitly Typing What this Must Be

TypeScript lets you add a special this parameter as the first parameter of a function to declare what type this must be when the function is called. This is a compile-time only feature — it disappears in the compiled JavaScript output. It lets TypeScript catch incorrect this usage at build time.

// Without this parameter — TS doesn't know what 'this' is
function greet() {
  console.log(this.name); // TS error in strict mode: 'this' implicitly has type 'any'
}

// With this parameter — TS now knows the required shape
interface HasName { name: string }

function greet(this: HasName) {
  console.log(this.name); // ✓ TS knows this.name is a string
}

const user = { name: "Arjun" };
greet.call(user);       // ✓ user has .name
greet.call({ age: 25 }); // TS Error: { age: number } has no .name

// this parameter is erased in compiled JS — it's purely for TS checking
11 — TypeScript Interview Q&A
Interview TypeScript questions on this topic
How does TypeScript improve safety around this?
In two key ways. First, TypeScript enforces strict mode by default — unsafe default binding gives you undefined (loud crash) instead of the global object (silent bug). Second, inside classes TypeScript fully types this — every property access through it is type-checked. And with the this parameter annotation on standalone functions, TypeScript can verify the correct object is passed when calling with call/apply/bind.
What is the arrow function class field pattern and when do you use it?
It's declaring a class method as an arrow function assigned to a class field: handleClick = () => { this.count++ }. Because it's an arrow function, this is lexically bound to the class instance — it can never be lost no matter how the method is passed around. Use it for event handlers and callbacks that need to reference this. It's the modern alternative to this.method = this.method.bind(this) in the constructor.
What is the this parameter in TypeScript?
It's a fake first parameter you can add to a function signature — function fn(this: SomeType, ...) — to tell TypeScript what type this must be when the function is called. TypeScript uses this to type-check call/apply/bind usage. The parameter is erased completely in the compiled JavaScript output — it has no runtime effect.
12 — Key Takeaways

JavaScript + TypeScript — What to Lock In

  • this is determined at call time, not at write time. Same function, different caller, different this.
  • Four rules, in priority order: new > explicit (call/apply/bind) > implicit (obj.fn()) > default (plain fn()).
  • Default binding gives window in non-strict, undefined in strict mode. TypeScript enforces strict mode.
  • Implicit binding loss is the most common this bug — extracting a method or passing it as a callback severs the object connection.
  • call = run now, individual args. apply = run now, array args. bind = return new function, this locked permanently.
  • Arrow functions have no own this — they inherit it lexically from the enclosing scope. call/apply/bind have no effect on their this.
  • Use arrow functions for callbacks inside methods. Use method shorthand for methods themselves. Never mix them up.
  • In TypeScript classes, use arrow function class fields for event handlers and callbacks — eliminates all binding loss risk.
  • The TypeScript this parameter lets you enforce what type this must be — erased at compile time, purely for type safety.
